# 我的世界服务器指北

## 引言

### 定位

本文档旨在为想要进入服务器开发行业的新人提供思想和技术上的帮助，同时总结笔者数年的服务器开发经验与找出笔者因基础不牢而产生的各种错误理解，属于技术书类。

本文档被设计为可供任何起点水平——包括基本没有接触过电脑——的新人阅读，故将涵盖较广范围，包括但不限于：电脑操作、网络技术、编程、Linux 系统操作、游戏学和美术类知识和技能。

本文档全篇使用 Markdown 语法编写，目的是便于在网络中传播以及便于维护。

本文提到的 Minecraft 若无特殊说明则均为 Java Edition 版本，与 Bedrock Edition 以及网易代理的电子游戏《我的世界》无关。

### 起因

**Minecraft** 这款游戏在笔者这一代人中红极一时，陪伴很多人长大。但在 2023 年的现在，其在中国国内的名气和玩家数都在持续下降。笔者时常就游戏本身询问身边人，但得到的回答大多都是「曾经玩过」云云。

在游戏本身冷清的当下，仍坚信它具有的潜力且有足够热情的玩家们都在摸索着自己的「救亡图存」之道路：有些人从游戏本身内容下手，无偿制作地图、模组等供它人游玩，而有些人则瞄准这款游戏概念上的优势，试图将其与教育或游戏制作结合，发挥其作为教育工具和游戏引擎（Engine）的作用。笔者属于后者，故撰写本文档为想要制作游戏却又没有足够的美术能力、理解能力或担心无法吸引玩家的人提供一条新的方向，同时也为想要为自己搭建实践学习的内容的舞台的学生们开拓一种新的可能性。

### 目标

本文档将以训练作为读者的你成为可以独立自主地设计、开发和运营一台完善的群组服务器为终极目标。本文大体也按照正常的开服流程组织，同时涉及一些游戏学和编程知识，以便往后的知识迁移。

中国的游戏市场以混乱著称，游戏机禁令时代的余波仍在影响着今天的玩家和市场本身，玩家素质和电子游戏本身的必要性等等问题至今仍未被解决。在笔者看来，我们游戏产业并不缺少技术过硬的大佬，而是缺少对游戏抱有热情的、愿意为之奋斗的、在心底里并不承认「电子鸦片」论的、相信电子游戏可以为社会做出贡献的挑战者（相关论述见 [附录 1](#)）。

故本文档也以思想启迪为目标，内含大量笔者对 Minecraft 乃至电子游戏本身的看法观点，希望能为我国电子游戏产业出一份微薄之力，推动电子游戏艺术的进一步发展。

## 第一章 Minecraft 与你

在本章中，笔者将对 Minecraft Java Edition 进行概念上的剖析，便于读者对这款游戏有更深入的理解。

### Minecraft 是什么

Minecraft 是一款瑞典公司 Mojang Studios（微软旗下）开发的沙盒游戏（Sandbox）。所谓沙盒游戏，一种说法是其名称取自儿童玩耍的沙池。孩子们可以在沙池中自由地堆砌建筑、修改地形和挖掘宝藏。沙盒游戏自然也以极高的自由度和**非线性**的游戏流程为主要特点。玩家可以在其中自由生存并修改世界本身，同时一般不被硬性要求完成某个任务（所谓非线性）。

Minecraft 就是一款典型的沙盒游戏，其世界以方块（Block）为最小单位构建。玩家可以在近乎无限大的世界中挖掘（Mine）和放置每一个方块、合成（Craft）各种物品、强化自己的武装、建造自己的小镇，抑或是挑战强大的 BOSS 等等。

在作为沙盒游戏的本体提供的自由度之上，游戏运营公司对于**社区**的高关注度和高开放度则为 Minecraft 成为沙盒游戏中的典范和不朽之作奠定了基础。玩家可以籍由命令（Command）、资源包（Resource Pack）、插件（Plugin）、模组（Mod）和数据包（Datapack）等等「游戏之外」的工具拓展游戏本体的内容，使 Minecraft 改头换面，甚至完全成为 [另一个游戏](https://terrariacraft.com/)。而这些丰富的内容，其中绝大部分都由社区中的作者免费编写并开源，提供给全世界的玩家自由选择和游玩。正是这无私的社区为 Minecraft 提供了源源不断的活力，使其在今天仍在世界市场上占有一席之地，被全世界的玩家们喜爱。

让我们在此感谢前人的奉献，使这篇文档有了在此被编写的可能。

笔者便是看中了其极高的可拓展性和丰富的可能性，并在此探索将其用做教育和游戏引擎的可能性。

### Minecraft 怎么来

既然是一款游戏，其当然免不了下载和安装。

Minecraft 以一种独特的方式分发其游戏本体。对于市面上的大多数游戏，游戏本体被打包为一个一键式安装包（Install pack），玩家点点鼠标，便可以将组成游戏本体的各种文件提取并安装到电脑中，再通过某个 .exe 格式的文件运行。但对于 Minecraft，则是将不同版本的游戏进行单独打包，玩家通过启动器（Launcher）下载想要游玩的版本（Version），并在启动器内开启游戏。

这种分发方法有许多好处，例如：

- 将游戏的各个版本隔离出来，以便玩家切换游戏版本以及自行拓展游戏内容
- 使拓展游戏本体的内容（模组等）不会因游戏版本的更新而失去作用，玩家仍可以通过下载老版本来使用这些内容

可以说，这种独特的分发方式也是 Minecraft 得以广泛流行和持续更新的重要原因。

所以，想要游玩 Minecraft，我们只需要下载一个启动器就好了。

市面上有各种各样的启动器，例如：官方提供的官方启动器、老牌的 HMCL（Hello Minecraft! Launcher）启动器以及新潮的 PCL（Plain Craft Launcher）启动器等。下文将介绍 PCL 启动器的使用方法。

### 启动器

PCL 启动器为国人作者 [龙腾猫跃](https://afdian.net/a/LTCat) 独立制作，这里是其在 MCBBS 上的 [发布页](https://www.mcbbs.net/thread-719579-1-1.html)。PCL2 采取免费与付费并行的制度，付费版可以额外获得快照版（Snapshot）的使用权限以及一些与启动器外观有关的定制功能（修改颜色等），但不付费也不影响主要功能的使用。

下载到启动器后，我们会得到一个名为 Plain Craft Launcher 2.exe 的文件（文件名可能有微小的变化）。将其放到你喜欢的某个文件夹中，如 D:/Minecraft/ 下（最好是子文件夹下，因为之后会有额外的文件产生）并点击运行，启动器就已经准备好为你下载和运行游戏了。

点击运行后，与启动器文件同级目录下生成了 PCL 以及 .minecraft 两个新的文件夹，前者中储存着与启动器相关的设置和部分账户信息，而后者则储存着所有游戏相关的信息，地图文件、截图文件等都可以在这里找到。

进入启动器后，我们可以先更改一项设置：

```txt
设置 - 游戏 - 启动选项 - 版本隔离：修改为隔离所有版本
```

这会将不同版本的游戏文件完全分离，防止其互相干扰。你可以在 **.minecraft/versions/** 目录下找到各个版本的游戏文件。

接下来，我们点击下载按钮，来到下载界面，选择你想要的游戏版本（可以是最新版），点击开始安装，PCL 将帮助你完成剩下的工作。

### Java 与你与 Minecraft

事实上，PCL 不止能帮你下载并安装游戏本体的打包文件。

顾名思义，Minecraft Java Edition 是一款由 Java 编写的游戏。所谓 Java，就是指一种编程语言，它是由 Oracle（甲骨文）公司开发的一门健壮且灵活的，面向对象的语言，在各种领域得到广泛应用。目前主要应用与后端开发、电子游戏开发和安卓应用开发中。

所有 Java 程序都需要在 Java 程序运行环境（Java Runtime Environment）中才能运行，Minecraft 也不例外。所谓环境，主要包括编译（Compile）环境和运行环境（虚拟机）。前者用于将人类书写的高级编程语言翻译为以 .class 为格式的字节码语言文件，而后者则负责依据不同的平台动态地将字节码语言翻译为电脑能理解的机器语言以执行。比起直接将高级语言翻译为机器语言，用字节码作为中继提高了 Java 的跨平台性，同时也保留了预先编译所提供的高效性。

我们可以通过在电脑中下载 JDK（Java Development Kit）以建立 Java 程序运行环境。所谓 JDK，指 Java 开发工具包，其中包含了上文所提到的编译器、虚拟机和各种调试工具等。各大公司都推出了自己开发的 JDK，包括 Oracle 的官方 JDK、OpenJDK 以及阿里巴巴开发的 Dragonwell JDK 等等，每个 JDK 都有自己的优化方式，本文以安装 Oracle JDK 为例：前往 Oracle 的 [官方页面](https://www.oracle.com/hk/java/technologies/downloads/#java17) 下载供 Windows 系统使用的 JDK 17 的 64 位 .exe 安装包（x64 Installer）并点击运行（请暂时记住安装路径），JDK 便已经被下载到你的电脑中了。我们通常说的「在游戏前要进行 Java 安装」便是指这一过程。

这时，打开你的 PCL 启动器，进入 设置 - 启动选项 - 游戏 Java，点击下方的自动搜索，你刚刚安装的 JDK 应该已经被启动器识别到了。若没有识别到，你也可以点击手动导入，导航到 JDK 安装目录下的 bin 目录中，点击 javaw.exe 文件，即可手动导入 JDK。

理所当然地，Minecraft 的大部分程序类扩展内容（插件、模组等）也都是由 Java 编写的，在后文中，我们也将开启对 Java 语言的学习。

话说回来，如果你的 PCL 启动器在启动游戏时检测到你并没有安装 JDK，它便会跳出弹窗并自动帮你选择合适的 JDK 版本并进行安装，这也是第三方启动器比起官方启动器的优势之一。还有很多便捷的功能也都会在后文中提及。

### Minecraft 的购买

你知道 Minecraft 是需要购买的吗？或者说，你真的需要购买 Minecraft 吗？

正如上文所说，Minecraft Java Edition 采取将不同的版本单独打包的方式分发游戏本体。这里所谓的「包」，即是指上文提到的由大量 .class 为格式的字节码语言文件组成的，以 .jar 为格式的集合（在概念上类似 .zip 压缩文件包，一个包中内含多个文件）。

但是，这个将包下载到电脑中的过程实际上并没有进行权限检查。也就是说，就算你并没有购买 Minecraft，也可以不受任何影响地从官方渠道下载各个版本的游戏并用启动器启动它们（在游戏内也不会进行帐号登录等操作）。这种游玩方式便是我们常说的「离线版」或「盗版」（不像一般游戏的盗版都对游戏本体经过修改，Minecraft 的盗版实际上仍使用官方的游戏文件游玩，所以此处的盗版是一种蔑称）。

为什么 Mojang Studios 选择如此开放的方式发布自己的游戏，我们不得而知，但可能的原因如下：

- 采用 .jar 包的不鉴权分发可以使 Mojang 在开发和发布游戏时更加方便和高效
- Mojang 想要鼓励玩家和开发者自由地创造和共享自己的插件、模组等内容，以丰富游戏的体验
- 一些人认为采用 .jar 包的不鉴权分发可以扩大 Minecraft 的用户基础以丰富社区，并吸引更多人购买正版

这种「鉴权」仅在一个环节中进行，那就是当玩家进入开启了正版验证的服务器时。这也意味着，没有购买游戏的离线版玩家并不能进入这些服务器中。具体内容将在下文中提到。这同时也是离线版与正版在游戏内唯一的区别。

当然，这并不意味着作为玩家的我们可以放肆地享受官方的慷慨，若是喜欢这个游戏并愿意支持它，请通过官方渠道购买游戏以支持其往后的开发。作为玩家的我们需要牢记，电子游戏并不是由神创造的或者被一阵风刮来的，你当下从其中获取的快乐都建立在它背后的开发者们无数小时的努力上。

### Minecraft 启动

言归正传，回到 PCL 启动器中，在启动游戏前，我们需要先进行帐号的注册：

- 若你购买的正版，便可以点击 启动 - 正版 按钮，并登录你的微软帐号以使用正版用户名游玩
- 若你没有购买正版，则要点击 启动 - 离线 按钮，并在 游戏用户名 框内输入你喜欢的游戏内名称（仅可包含数字、字母和下划线_）即可用离线用户名游玩

设置完成后，点击启动游戏，若不出意外，Minecraft 应该已经运行在你的电脑中了。

虽然并不强求，但在继续向下阅读前，仍建议读者先花时间熟悉 Minecraft 游戏本身，便于对之后章节的理解。

## 第二章 千变万化的 Minecraft

本章将暂时抛开服务器不谈，简单介绍在玩家层面如何充分发挥 Minecraft 作为沙盒游戏的巨大潜力。

### 多彩的原因

作为沙盒类的代表作，Minecraft 正如前文所说是一款由社区主导的自由的游戏。玩家可以凭借自己的喜好对游戏本体或游戏内容进行修改并共享修改的成果。

因此，社区成员以开放和共享为信条，在 Minecraft 这款游戏搭建的舞台上自由地起舞，创造一个又一个新的内容。

### 社区在哪

玩家聚集在一起交流经验、共享资源，社区便在这一合作过程中自然而然地被建立了。玩家需要资源，而社区提供资源，享受社区带来福利的玩家又自发地回报社区，为社区做自己的贡献，Minecraft 便建立在这种良性循环之上。

社区多以网站论坛为载体，以下列出一些有名的玩家社区：

- [MCBBS](https://www.mcbbs.net/)：又名我的世界中文论坛，是国内最大的玩家论坛，其中各种资源包罗万象
- [MCMod](https://www.mcmod.cn/)：又名我的世界模组百科，是国内最大的专精模组发布的平台
- [CurseForge](https://www.curseforge.com/)：数个游戏的共同社区，主要由模组开发者构成，是 Minecraft 模组最主要的发布地点
- [Planet Minecraft](https://www.planetminecraft.com/)：国外老牌社区，主要由地图作者、资源包作者和光影作者组成
- [Spigot](https://www.spigotmc.org/)：著名服务端 Spigot 的官方论坛，是外国服务端插件的主要发布地点
- [Modrinth](https://modrinth.com/)：新兴的资源社区，提供的内容与 CurseForge 类似，都包罗万象
- [BuiltByBit](https://builtbybit.com/)：老牌的资源市场（旧称 MC Market），售卖多种类型的资源
- [PolyMart](https://polymart.org/)：新兴的资源市场，售卖多种类型的资源

### 如何起舞

上文中笔者不断强调 Minecraft 的多姿多彩，现在便是关注细节的时候了。

扩展游戏内容的方法有很多，主要是以下几个：

#### 地图

Minecraft 以世界（World，通常也被称为地图）为单位储存游戏数据，在单人游戏新建世界（也就是新建了一个地图）后，玩家对世界进行的所有修改，包括放置的方块、击杀的生物、身穿的装备等等所有的数据都被储存在 .minecraft 文件夹的地图文件中。

你可以在 **.minecraft/versions/你的游戏版本名称/saves/** 目录下找到你创建的世界文件。若你想将自己游玩的世界分享给他人，仅将这个文件发送给他即可。

至于社区中的地图，实际上就是经地图作者们修改过的世界。地图作者通过命令方块和数据包等手段为世界添加各种新建筑、新功能、添加新物品，如枪械等，并将世界文件分享给其他玩家游玩。

有名的大型地图有很多，如著名 RPG 游戏地图 [盘灵古域](https://pan-gu-continent.blogspot.com/) 等。你也可以在上文提到的 MCBBS、Planet Minecraft 等社区中找到他人制作的各种地图进行游玩，仅需将下载到的世界文件放入上文提到的世界文件 saves 目录中即可在游戏中看到它。

#### 资源包

资源包（Resourse Pack，旧称材质包）是 Minecraft 为玩家提供的便捷地覆盖游戏原版纹理的方式。简单地说，通过安装资源包，玩家可以修改游戏本体的外观方面的内容，包括但不限于：方块外观、图形界面外观、生物外观、游戏内文字和音效等。

资源包的安装十分简单，仅需将下载到的资源包（大多以 .zip 格式结尾，是一个压缩文件）放入 .minecraft/versions/你的游戏版本名称/resourcepacks/ 目录下即可在游戏内找到。

著名的资源包有很多，例如 [Chocolate bars pack](https://www.planetminecraft.com/texture-pack/1-10-1-12-16x-chocolate-bars-pack/)。

在后文中，读者也需学习资源包的制作方法。

#### 光影包

光影包（Shader Pack）的定位与资源包相似，只不过其修改的是 Minecraft 原版的光照相关的内容，如阴影、反射和动态光照等等。

与资源包不同的是，官方并没有给出直接修改游戏本体光影数据的方式，这也意味着，玩家需要在游戏本体之上安装特殊的光影加载器（Loader）才能使用他人制作的光影。

目前主流的光影加载器有 Iris（Fabric 平台）和 Optifine（Forge 平台、原版）。由于光影加载器属于纯客户端内容，故本文不深入讲解，以下是一些介绍视频：

著名的光影包有很多，例如 [BSL Shaders](https://www.curseforge.com/minecraft/shaders/bsl-shaders)。

#### 模组

模组（Mod）应该是老玩家耳熟能详的内容。通过模组加载器，玩家可以籍由他人编写的模组包（.jar 格式的 Java 程序包）修改游戏本体的代码，为游戏添加新的物品、生物、界面、维度和方块等，而这也是只能在原版框架内起舞的地图作者们望而却步的内容。

模组与光影包同样，是不能直接被 Minecraft 原版游戏加载的。我们需要安装模组加载器才能使用模组。目前主流的模组加载器主要是 Fabric 和 Forge 两个。在 PCL 启动器中下载游戏时，可以直接通过点选的方式安装这些模组加载器。

需要注意的是：

- 多个模组加载器在绝大多数情况下不能共存在同一个 version（游戏）中
- 模组编写时便已经与模组加载器绑定，这意味着在 Fabric 平台编写的模组不能被安装到 Forge 环境中
- 模组编写时也与游戏本身的版本严格绑定，这意味着你不能在 1.20.2 版本的游戏中安装一个在 1.7.10 版本下编写的模组
- 模组与模组之间也会出现不能共存的情况

在选择模组加载器和安装模组时，玩家需要关注以上四点，做到平台 - 版本 - 模组三者对应。

部分作者通过将大量模组同时安装并进行一些修改使其融合，就形成了整合包。通过下载和安装这些模组的集合，玩家可以体验到完全不同的 Minecraft。如大多数整合包都会引入原版没有的任务（Quest）系统等。以下是一些著名的整合包：

- [水星迫降重制版](https://www.mcbbs.net/thread-1097781-1-1.html)：以科技类模组构成的整合包
- [永恒乐事](https://www.bilibili.com/video/BV1Ws4y1B7yw)：大型烹饪向整合包
- [异界生活幻想](https://www.bilibili.com/video/BV1tU4y1A7km)：大型冒险向整合包

而所谓模组服，就是在服务端中运行这些模组的集合，并提供给多个玩家同时游玩。在之后的文章中，读者将会学习到模组服务器的开设方法。

#### 数据包

数据包（Data Pack）与模组类似，是 Minecraft 官方提供的一种修改游戏内容的方式。与模组不同的是，数据包并不直接修改游戏代码，这意味着数据包比模组更加轻量便捷，但也不像模组一样无所不能。同时数据包也不是 .jar 格式的 Java 程序包，而是以名为 JSON 的轻量级数据交换语言编写的。

目前，数据包主要用于修改战利品表（Loot Table）、合成配方（Recipe）、进度（Advancement，旧称成就）、结构（Structure，又称建筑）和世界生成器（World Generator）等内容。

玩家将下载的数据包（.zip 格式的压缩文件）放入 .minecraft/versions/你的游戏版本名称/datapacks/ 目录下，即可在游戏内使用 /datapack 命令启用它。注意服务端中安装数据包的位置略有不同。

著名的数据包有很多，如 [Terralith](https://www.planetminecraft.com/data-pack/terralith-overworld-evolved-100-biomes-caves-and-more/) 系列，对主世界（Overworld）、下界（Nether，旧称地狱）和末地（End）的世界生成器进行了大范围美化。

由于应用范围不广且仍有待完善，故本文档不涉及数据包编写知识。

#### 总结

总的来说，通过以上这些多种多样的资源，玩家可以扩展自己的游戏内容，发挥创造力，从而强化和延申原版游戏的内容，抑或是将其全部推翻，创造一个新的游戏。作为一个普通玩家，他的大部分时间应该也都投入在与这些内容的交互中了。

## 第三章 成为服务者

在对 Minecraft 游戏本身和其扩展内容有了基本的认知后，读者将在本章学习服务器相关的知识。技术性内容也将在本章开始被引入。若遇到无法理解、且又没有在后文被解释的专有名词，请善用谷歌等搜索引擎。

### 何为服务

在上文中，笔者介绍的无一例外都是关于「玩家」的事。下载游戏、安装游戏、启动游戏、下载和安装地图、模组、资源包、光影包、数据包，这些操作均是在玩家自己的电脑中进行的，而也只有玩家本人才能享受到这些操作带来的乐趣，这时玩家本人可以说是「体验者」。

那么自然而然地，「服务者」便出现了。与为自己下载游戏的个人玩家相对，服务者们不仅仅为自己，更是为其他玩家搭建游戏的平台并允许多个玩家同时在一个 Minecraft 世界内游玩。这时，这种开放给所有玩家游玩的 Minecraft 便被称为服务器，而这些服务器的作者便是我们。

在 PCL 启动器为你启动游戏的过程中，实际上是运行了一个以 .jar 格式结尾的 Java 程序包。这个程序包可以在 ./minecraft/versions/你的游戏版本名称/ 目录下找到（目录下的其他文件都可以被理解为供这个程序包使用的资源文件，如图像、音频文件和配置文件等），根据游戏版本不同，文件名会有差异，如 1.20.2.jar（1.20.2 纯净版）、1.20.1-Fabric 0.14.22.jar（1.20.1 的 Fabric 模组加载器版本）。这时，这个正在运行的 Java 程序包在计算机术语中便被称为一个「服务」。所谓服务，可以向用户提供某种特定的功能或满足用户的某种需求，并通过网络或本地系统接口与其他应用程序或用户进行通信。

显然，在本地启动的 Minecraft 游戏这一语境中，服务便指的是上文提到的那个 Minecraft 的以 .jar 为格式的程序包所包含的程序，而其提供的功能则是我们正在游玩的游戏本身，用户则指坐在电脑前的读者本人。这个游戏程序包也被称为客户端，因为它只能向一个单独的用户提供它的游戏服务。而这种仅供一个用户（本地用户）使用的服务也被称为本地服务。

### 怎么服务

在上文中，我们了解到，游戏官方为玩家分发的游戏本体实际上就是一个以 .jar 为格式的 Java 程序包。在被运行后，这个程序包只能向一个本地玩家提供游戏服务。

那么，有没有一款 Minecraft 的 .jar 程序包被设计为可以向多个用户提供服务呢？

这就是官方所提供的 Minecraft Server 程序包（又称 Minecraft 服务端，与客户端相对）。它也一样以 .jar 为格式，与客户端 .jar 程序包一样可以被 Java 虚拟机运行在一台电脑上。而这种向多个用户提供服务的电脑便被成为服务器，意思是提供服务的机器，而运行 Minecraft Server 程序包的机器便也被成为 Minecraft 服务器，意为向多人提供 Minecraft 游戏服务的机器。

与 PCL 启动器帮助我们启动 Minecraft 客户端程序包类似，我们也可以通过简单的流程启动我们的 Minecraft 服务端程序包。

#### 下载服务端

前往 Minecraft 官网中的 [服务端介绍页面](https://www.minecraft.net/en-us/download/server)，我们可以找到游戏最新版本（1.20.2）对应的服务端的下载按钮。

点击下载后，我们便可以得到一个名为 server.jar 的程序包。这便是所谓的服务端程序包。我们将它重命名为 minecraft_server.1.20.2.jar，以提示我们它是一个 Minecraft 服务端程序包，以及它对应的游戏版本是 1.20.2。

#### 启动服务端

类似客户端游戏拥有一个 PCL 启动器可以帮助我们启动程序包，服务端游戏也有这样的启动器。但作为初学者，我们需要深究本质，使用运行批处理命令 .bat 文件的方式来启动我们的服务端程序包。

将刚刚下载到的 minecraft_server.1.20.2.jar 程序包放入一个单独的空文件夹中（本例中为 FirstServer 文件夹，指这是我开启的第一个游戏服务端），这个文件夹在今后将容纳这个服务端所有的数据文件，是你游戏服务端的「本体」。

接着，在这个文件夹下新建一个 .txt 文件（文本文件），并将以下内容写入：

```bat
@ECHO OFF
"C:\Program Files\Java\jdk-17\bin\java.exe" -Xms1024M -Xmx2048M -jar minecraft_server.1.20.2.jar --nogui
pause
:: 其中 "xxx\bin\java.exe" 中的 xxx 是前文安装 Java 时读者记忆的 Java 安装地址。
:: 这样写明 Java 的安装路径，目的是在运行 Java 程序包时指定 Java 的具体版本
```

接着，保存文件并将它的扩展名从 .txt 修改为 .bat，再给它起一个喜欢的文件名（本例为 .start.bat，文件名开头的符号 . 的作用为让这个文件始终处于文件夹的上方），你就新建了一个批处理文件。

所谓批处理文件，是 Windows 系统的脚本文件。而所谓脚本，则是帮助用户快速地执行一系列命令和操作的文件。

上面的这个 .start.bat 批处理文件做了一件简单的事情，即为运行了以下命令：

```bat
java -Xms1G -Xmx2G -jar minecraft_server.1.20.2.jar --nogui
```

这条命令启动了一个 Java 虚拟机（理解为一台虚拟的电脑），并在其中启动了刚刚下载的 Minecraft 服务端程序包。

至于 -Xms -Xmx 则是命令参数，决定这个虚拟机可以占用多少电脑的运行内存（本例最大为 2G，最小为 1G）。关于启动命令的参数的知识将会在优化相关章节中详细提及。

双击 .bat 文件，若不出意外，你的屏幕上应该出现了一个黑色的窗口（Cmd，译为为命令行，可以在其中输入命令来与服务端程序交互，如停止服务端），其中快速跳出多条日志信息（Log，指在程序运行过程中输出的记录性信息，内容包括无害的提示信息、警告消息、错误提示和关键事件记录等）。之后，你的 FirstServer 文件夹中应该生成了包括 eula.txt 在内的多个文件和文件夹，接着，你的黑色窗口处应该跳出了一个红色（或正常白色）的错误和一条日志信息：

```txt
You need to agree to the EULA in order to run the server. Go to eula.txt for more info.
```

接着，你会看到一段中文提示：

```txt
请按任意键继续...
```

当你看到这条中文提示时，一般意味着你的批处理命令因为一些特殊的原因终止了，并没有为你成功启动服务端程序包。至于具体原因是什么，让我们从上一条日志信息中找答案。这条英文提示翻译后如下：

```txt
你需要同意 EULA 才能运行服务端。前往 eula.txt 查看更多信息。
```

按照提示，我们打开 eula.txt 文件，看到其中内容如下：

```yml
#By changing the setting below to TRUE you are indicating your agreement to our EULA (https://aka.ms/MinecraftEULA).
#Mon Oct 09 18:32:35 CST 2023
eula=false
```

译文如下：

```yml
#通过将下面的设置的值修改为 true 以声明你同意了 EULA (https://aka.ms/MinecraftEULA).
#Mon Oct 09 18:32:35 CST 2023（第一次启动服务端的时间）
eula=false
```

这里的 EULA（End User License Agreement）是 最终用户许可协议 的简称。很显然，就像是任何软件服务在被提供前一样，官方制定了一些守则，你需要同意这些规章制度，才能合法地使用软件本身。

关于这个 EULA 以及 Mojang Studios 和微软公司发布的几篇守则的具体内容，笔者将在后文服务器设计相关章节具体介绍。现在，读者只需要听从指示，将设置修改为 true 即可，像这样：

```yml
eula=true
```

此处的 true 和 false 是布尔值（Boolean），其含义可以简单理解为 是 或 否。通过将 eula 这项设置的值从 false 修改为 true，就表示我们同意了这个协议。这个过程被称为修改配置文件，这时 eula.txt 就是配置文件。服务端程序包在启动时会检测这个设置的值，若不为 true，就会像刚刚那样输出错误信息并提示你查看这个文件。

修改完毕并保存后，再次点击 .start.bat 批处理文件，服务端应该被顺利地启动了（当你看到 Done (40.850s)! For help, type "help" 这条日志时，你就成功启动了服务端）。

#### 进入服务端

上文提到，服务是以网络的形式向用户提供需要的信息和交互的。我们想要使用服务端提供的游戏服务，就要在庞大的互联网世界中找到正在提供服务的那台电脑并与其建立连接。这有点像在一个大城市的一栋宾馆中找到自己的友人，我们需要知道这个宾馆的地址在哪。

对于互联网这个大城市，我们通过 IP 地址 以访问互联网中的某一台电脑设备和其提供的服务。你有时候会看到的：

```txt
192.168.2.1
127.0.0.1
40.52.17.136
```

这些以三个英文句点分隔的四个数字组成的字符串都是一个 IP 地址。每个电脑设备都被分配了自己独特的的 IP 地址。通过这些地址，我们的电脑可以在互联网中定位到一台具体的设备。

但只是定位到设备还不够。若是想在宾馆中见到朋友，你还需要知道它住在哪个房间里。

电脑也是同样。一台电脑不可能只提供一项服务，正如一个大房子里不可能只住一个人。电脑将每个服务都装在一个单独的小房子里，并用端口号（Port）进行标识，就像宾馆的每个房间都有门牌号。

于是，我们便用地址 + 门牌号，也就是 IP 地址 + 端口号 的方式，在庞大的互联网中定位到了需要的游戏服务，具体写法如下：

```txt
127.0.0.1:8080
192.168.2.1:22
40.52.17.136:25565
```

那么，如何知道我们正在运行游戏服务端的电脑的 IP 地址，以及这个服务究竟通过哪个端口提供数据和信息呢？

对于前者，我们可以简单的使用 127.0.0.1 这一 IP 地址来代表我们的电脑。这个 IP 地址代表「本地」（或者称本机）。由于我们正在运行服务端的这一台电脑上试图使用其提供的服务，也就相当于在朋友的房间里找朋友。无需其他定位，因为朋友就在我们面前。因为这个 IP 地址用于定位本机，所以有时也可以用 localhost 代替（译为本地主机）。

而对于后者，我们可以在服务端文件夹中的配置文件 server.properties 中找到服务使用的端口（默认为 25565）：

```yml
server-port=25565
```

你也可以将它修改为其他的值，之后关闭服务器（在命令行中输入 stop 并回车即可），再重新启动时端口就会更改。

言归正传，想要加入运行在本地上的服务端游戏，我们只需在 PCL 中启动游戏，点击多人游戏，点击添加服务器，在 IP 地址一栏中输入这个服务所在的的地址，也就是：

```txt
127.0.0.1:25565
```

或

```txt
localhost:25565
```

之后点击确认，你的服务器展示条（Motd）应该已经显示在游戏中了。双击这个展示条，你一应该已经成功进入了游戏，开始与服务端进行信息交流，享受它所提供的服务了。

若你发现自己无法进入服务器，那么请想想前文曾提及的，正版玩家和离线版玩家在游戏内唯一的区别，即离线版玩家无法加入开启了正版验证的服务器。所以，假如你没有购买正版，你便需要将服务器的正版验证关闭。你可以在服务端文件夹中的配置文件 **server.properties** 中找到如下配置：

```yml
online-mode=true
```

这代表正版验证已开启。将它设置为 false 并重启服务器，你现在应该可以使用离线版账号进入服务器了。

### 究竟怎么服务

在上一节中，我们在自己本机上启动了服务端并用本机上的游戏连接并进入了服务端。这时有些读者提出了质疑：这难道不还是和客户端游戏一样，只能一个人自己玩吗？究竟怎么才能让其他人也进来玩呢？

其实，在上一节中，我们已经讲完了基本原理。其他的玩家只需要有样学样，获得你服务端所在机器的 IP 地址以及服务所在的端口号，即可像你本机一样轻松地连接到游戏内了。

所谓「其他人」是否能访问到你本机的服务，取决于你本机（服务端）和其他人（客户端）所在的网络环境。

本文档不是互联网知识教学文章，所以仅作简要说明：对于一般的家庭电脑，都以路由器为网络连接的中心设备。路由器为保护你设备中的信息不被外界窃取，会通过一种转换技术将你家庭网络中的设备及其地址隐藏在一个特定的内部网络体系中，这样，你的家庭网络（局域网的一种）中的设备将无法被外部的设备（家庭网络之外的设备）访问到，而你家庭网络访问外部网络的请求则会被路由器通过转发而正常发送。

这么说来，假如我们有两台电脑，且两台电脑连接到同一个家庭网络中（连接到同一个路由器上），那么只要知道作为服务端的那台电脑在家庭网络内部中的地址，就可以用另一台电脑访问作为服务端的电脑上的游戏服务了。

在 Windows 系统中，我们可以使用命令获取本机在家庭网络中的地址：使用快捷键 Win + R 打开运行菜单，在其中输入 cmd 并回车，进入命令行。在其中输入 ipconfig 命令并回车，显示的信息中以 192.168 开头的 IPv4 地址就是本机在家庭局域网中的 IP 地址。

之后，你可以在另一台设备上测试是否能使用此 IP 地址 + 端口号访问服务端设备上的游戏服务。

那么，要如何才能突破这种网络上的限制，让全国乃至全世界的玩家都可以访问自己的服务器呢？

这个问题的本质是如何解决网络限制的问题。主流的解决办法有以下几个：

1. 换一台处于可以被所有人访问的网络环境下的设备
2. 在本机中部署内网穿透服务，让外部可以通过特殊的 IP 地址访问内部网络
3. 找自己的网络运营商购买一个动态的公网 IP 地址并将其解析到域名上，外部可以通过这个 IP 地址访问内部网络

本文将会采用方法 1 的思路，介绍使用云服务器 VPS 的解决方案。但现在让我们离开网络知识，回到服务端本身中。

### 扩展服务内容

上文中笔者曾强调了 Minecraft 这个游戏的多样性，玩家可以通过外部资源（模组等）扩展游戏本体的内容。现在我们开始学习架设游戏服务器，读者一定也产生了各种奇思妙想，想要创造属于自己的 Minecraft。问题在于，怎么才能将之前讲到的那些外部资源应用到我们的服务器中？

虽然有个别例外，但是服务器市场对游戏内容进行扩展的方法无非可以总结为两个：插件与模组。应用这两种方法的服务器也被分别称为插件服和模组服。

插件这个词是第一次正式被提及，它在表现上与模组相似，在此先介绍它们的相同点和区别：

- 相同点
  - 大多都是以 Java 语言编写的 .jar 程序包
  - 都可以向游戏本体添加新内容
  - 都有丰富的社区生态（指丰富的资源）
- 区别
  - 插件
    - 优势
      - 所有的插件服都不需要客户端安装任何多余的程序包即可游玩，对用户十分友好
      - 插件服相对模组服稳定性更高且占用的电脑资源较少（对性能要求低）
      - 插件服的开设更加简单方便
    - 劣势
      - 插件相对于模组，在编写方面更加受到原版框架的限制，难以直接实现部分功能，如：注册新实体、添加新方块、注册新物品和添加新图形界面等，只能通过折中和取巧的方法实现（与材质包相关，将在下文具体介绍）
      - 相对模组社区，插件社区的付费趋势更明显，有时插件服务器开发者需要为一些制作精良的插件付出一定的费用
  - 模组
    - 优势
      - 模组在技术上可以实现比插件更多的功能，且这些功能本身已有丰富的生态（插件虽然也做的到，但仍没有自己的暮色森林或者匠魂）。
      - 相对插件社区，模组社区仍以免费提供内容为主
    - 劣势
      - 大部分向游戏添加新内容的模组都要求在服务端和客户端同时被安装（客户端和服务端需要互相交换和处理那些新的信息，客户端也可能需要新的纹理资源），这带来了客户端难以被服务端开发者维护和控制的问题，用户可能需要反复下载你制作的新客户端整合包（模组的集合）
      - 模组服比插件服消耗更多的电脑资源
      - 模组服的开设比插件服更加困难
      - 模组的编写比插件更加困难
  - 原版
    - 优势：
      - 原版服务端拥有最强的稳定性
      - 开设使用原版服务端的服务器更不容易触犯 EULA 中的限制
      - 可以提供最纯净的游戏体验
    - 劣势
      - 除了命令和数据包之外没有可以扩展玩法内容的手段
      - 难以享受部分与安全性相关的插件或模组提供的服务

原版游戏程序包并没有提供加载这些插件或模组的渠道，于是一些有能力的作者们便制作了插件加载器和模组加载器，在原版程序包这个盒子上打出一个洞，并将制作的插件和模组注入到原版游戏中。这些与原版服务端不同的，被加入了插件或模组加载器的服务端，也被我们称为修改服务端。

修改服务端自然也分为插件和模组两派，以及试图让二者共存的混合派（同时加载插件和模组），其中代表性的服务端如下：

- 插件服务端
  - [Bukkit](https://dev.bukkit.org/)：最初的插件服务端，因违法 EULA 而被官方告上法院，被中断维护（很少被直接使用）
  - [Spigot](https://www.spigotmc.org/)：Bukkit 的继任者，插件社区生态的开创者（很少被直接使用）
  - [Paper](https://papermc.io/)：Spigot 的下游服务端，在 Spigot 之上引入了众多优化和新功能
  - [Purpur](https://purpurmc.org/)：Paper 的下游服务端，在 Paper 之上引入了众多其他服务端的优化代码

- 模组服务端
  - [Forge](https://files.minecraftforge.net/) 官方服务端，用于加载 Forge 模组（通过特殊的安装程序对原版程序包进行注入操作）
  - [Fabric](https://fabricmc.net/) 官方服务端，用于加载 Fabric 模组（直接提供注入完成的 .jar 服务端程序包）

- 混合服务端
  - [CatServer](https://catmc.org/)：老牌模组服务端，用于同时加载 Forge 模组和插件（提供修改完成的 .jar 服务端程序包）
  - [Arclight](https://github.com/IzzelAliz/Arclight)：新派模组服务端，用于同时加载 Forge 模组和插件（提供修改完成的 .jar 服务端程序包）
  - [Mohist](https://mohistmc.com/)：新派模组服务端，用于同时加载 Forge 模组和插件
  - [Cardboard](https://cardboardpowered.org/)：处于测试阶段的模组服务端，用于同时加载 Fabric 模组和插件（本身也是一个 Fabirc 模组）
  - [Sponge](https://spongepowered.org/)：服务端界的清流，在支持 Forge 模组加载的同时试图建立自己的新插件生态，不支持 Bukkit 及其子系的插件，而是新开辟了 Sponge 插件生态（仍不完善）

以上服务端只是众多服务端中的几种，市场上的服务端种类繁多，读者需要按需选择。混合服务端听起来两全其美，实际上兼容性和稳定性不佳。而纯插件或模组服务端则继承插件和模组本身的优劣。笔者在同时使用了三种服务端后最终选择纯插件服务端类中的 Purpur 服务端，但后文则考虑到先后性，选择先使用 Spigot 进行演示。至于 Purpur 则会在实践章节中提及。

## 第四章 插件服的开设

在上一章中，笔者介绍了服务端的种类并选择以纯插件服务端 Spigot 作为本章的演示服务端。
在本节中，笔者将介绍插件的安装方式和服务端的文件结构。

### 启动服务器

在第三章中，我们下载了官方提供的服务端并使用 .bat 批处理命令启动了它。但正如上文所说，官方提供的服务端无法加载任何插件或模组。所以，读者需要将服务端切换到 Spigot 服务端。

你不能直接在 Spigot 官网上下载到它的 .jar 服务端程序包，只能下载到它的 BuildTools（构建工具），用户需要自己通过构建工具向原版服务端进行 Spigot 插件加载器的注入。原因有些复杂，简单地说就是 Spigot 为了不走 Bukkit 的老路而选择让用户自己修改 Minecraft 原版游戏的源代码。

所以为了方便起见，读者可以前往第三方平台 [GetBukkit](https://getbukkit.org/) 获取注入完成的 Spigot 服务端。

下载完成之后，将得到的 spigot-1.12.2.jar 服务端程序包放入新的独立文件夹中（本例为 SecondServer），之后按照上文提到的方法制作一个 .bat 批处理文件，其内容如下：

```bat
@ECHO OFF
"C:\Program Files\Java\jdk-17\bin\java.exe" -Xms1024M -Xmx2048M -jar spigot-1.20.2.jar --nogui
pause
:: 别忘记将 "" 中的内容替换为自己的 Java 安装地址
```

之后按照先前的做法双击 .bat 文件启动服务端，之后同意 EULA 并再次启动，若在控制台中看到类似 Done! 的字样，你的插件服务端应该已经成功启动了。

### 连接服务器

与前文讲过的方法相同，你可以在本机使用：

```txt
127.0.0.1:25565 
```

这一地址在游戏中连接到你的服务器。

### 插件的安装和使用

#### 安装插件

上文已经提到过，所谓插件，与模组有些相似，都是一个可被服务端检测并运行的 .jar 格式的程序包。

想要安装插件，我们需要先下载一个插件。让我们从权限插件开始。

你可以在 Spigot 论坛中下载到丰富的插件（需要注册账号才能下载），其中就有著名的权限插件 [LuckPerms](https://www.spigotmc.org/resources/luckperms.28140/)。

下载完成后，我们会得到名为 LuckPerms-Bukkit-xxx.jar 的插件程序包（其中 xxx 是插件的版本号）。将这个插件放入 SecondServer/plugins/ 这个目录下，之后将服务器重启，你的 plugins 文件夹中应该多出了名为 LuckPerms 的新文件夹。

在每次服务端开启时，都会检测并运行 plugins 文件夹中的所有插件。大部分插件被首次加载后，都会生成一个与插件同名的文件夹，其中大多包含了若干个以 .yml 结尾的文件。

这个文件夹被称为插件的配置文件夹，其中储存了插件的数据（.db .json 之类格式结尾的文件大多用于储存插件需要的数据，如权限插件需要储存玩家都有哪些权限）和配置文件（大多以 .yml 结尾）。其中配置文件最为重要。

#### 配置插件

就像前文接触过的 eula.txt 一样，插件也需要配置文件让服务器开发者自由的控制插件的行为或表现。如部分插件会自带多语言支持，服务器所有者可以根据自己客户使用的语言来配置插件在游戏内发送的各种信息的显示语言。

.yml 格式是配置文件的常用格式，其对应的 Yaml 语言的特点是易读和易于扩展，以下是一个 yml 配置文件可能的样子：

```yml
# 储存团队信息的 yml 文件
my-team:
  name: 'Forest Development'
  team-members:
  - 'YK_DZ'
  - 'Cerf_volant'
  - 'Mamu_Numb'
  team-data:
    rank: 5
    income: 0
    has-dream: true
```

如你所见，这个 .yml 文件描述了一个名为 Forest Development 的团队，并用一个表格容纳了其中的成员，还储存了一些数字和布尔值。

Yaml 语言以键值对的方式储存信息。比如 name: 'Forest Development' 被称为一个键值对，name 是键，而 name 这个键被赋予了一个值，即 Forest Development。同理 team-members 也是一个键，其中储存了一个列表（包含所有成员的名字）。

至于 # 之后的内容则被称为注释，没有实际含义，一般用于告知用户某个键值对的作用。

大部分插件都没有实时加载配置文件的能力，当你修改了插件（服务端同理）的配置文件后，需要重启服务器或重载插件（大部分插件都有设计 reload 命令来重载自身的配置文件，如 LuckPerms 插件的命令是 /luckperms reloadconfig）来应用你的更改。

### 插件服务端结构

上文曾提到，服务端实际上指 spigot-1.20.2.jar 这一 Java 程序包，服务端文件夹中的其他各种文件都供服务端程序使用，如用于储存游戏数据或配置服务端行为。

部分这些文件和文件夹的具体作用如下：

- logs：储存服务器日志的文件夹，显示在控制台中的字符都会被储存到日志中。每次服务端重启后，上一次的日志就会被以压缩包的形式储存起来，而 latest.log 就是最新的日志信息
- plugins：储存服务端插件的文件夹，将插件放入其中以安装
- world & world_nether & world_the_end：储存世界（又称维度）数据的文件夹，分别是主世界、下界和末地。你也可以通过多世界插件在服务器中添加其他新的世界，将在后文插件服设计逻辑中提及
- bukkit.yml：Spigot 继承了 Bukkit 服务端的代码，所以也继承了 Bukkit 服务端本体的配置文件，其中内容将在附录中详细解释
- spigot.yml：Spigot 服务端本体的配置文件，其中内容将在附录中详细解释
- server.properties：服务器的基本配置文件，包含正版验证、服务器端口、是否允许命令方块等等常用配置

其他的文件则因为重要性较低而不特意提及。

### 常见的核心插件类型

现在，读者已经有能力独自安装并配置插件了。通过将数个、数十个甚至数百个插件进行组合并配置，你可以让游戏本体变成你想要的样子。现在，让我们进入设计环节，介绍插件服中几类典型且核心的插件和它们是怎么帮助我们设计游戏的。

#### 基础插件

所谓基础插件，一般指提供各类基础功能的插件，这些功能包括：

- 实用的命令及其缩写：如 ``/gm <0/1/2/3>`` 可以快速切换游戏模式、``/hat`` 可以将手上的物品戴在头上、``/tpa <玩家名>`` 可以让玩家申请传送到某个玩家等等
- 椅子功能：右键台阶来坐在上面
- AFK 功能：玩家一段时间不移动则将其标为挂机模式，使其免受伤害，或者将他们踢出服务器
- 地标功能：一般表现为 ``warp`` 命令打开一个菜单，玩家可以点击其中的按钮来传送到某个位置
- 背包检查：管理员可以通过命令打开玩家的背包并修改其中的内容
- 不胜枚举...

常见的基础插件则有：

- [EssentialX](https://www.spigotmc.org/resources/essentialsx.9089/)
- [CMI](https://www.spigotmc.org/resources/cmi-298-commands-insane-kits-portals-essentials-economy-mysql-sqlite-much-more.3742/)

基础插件一般都包含大量杂乱的功能，例如有名的 CMI 插件就包含超过 300 个简单的基础功能。

虽说被称为基础插件，但其并不是必须要安装。由于其包含的功能过多，所以也会产生额外的性能占用，同时也会留下大量用不到的功能。另外，功能种类繁多当然也就意味着每一种功能都不可能做的十分精细。如果你的服务器对某一个小功能要求较高，可能还需要为其暗转专门的插件，这显然是得不偿失的。

所以，你需要根据自己服务器的需求来选择是否要按安装基础插件，以及要安装哪个基础插件（请勿尝试安装两个基础插件，这种行为本身是多余的，且技术上可能会导致命令冲突）。

#### 权限插件

不像单人游戏一样，多人游戏中的每个人理所当然地会拥有不同的权限（Permission）。比如服务器管理员可以自由地调整自己的游戏模式，而服务器的玩家则不能。管理每个玩家可以做什么便是权限插件的责任。

每个插件都有自己的权限。比如 LuckPerms 插件就为自己的每个插件命令都设置了一个权限，若没有这个权限，玩家就不能执行这个命令。这些权限一般都会写在插件的发布页面或 WIKI 文档中。

这些权限一般以以下形式被命名：

```txt
minecraft.command.gamemode
cmi.command.tpa
quickshop.use
```

位于权限开头的是这个权限所属的插件，比如 minecraft（原版命令）、cmi（名为 CMI 的插件的命令）和 quickshop（名为 QuickShop 的插件的命令）等。而之后的内容则解释了这个权限具体控制的内容，如 minecraft.command.gamemode 控制玩家是否可以使用原版的 gamemode 命令切换游戏模式。而 quickshop.use 则控制玩家是否可以使用插件 QuickShop 提供的箱子商店功能。权限具体的作用以插件发布者的说明为准。

另外，你还可以使用通配符 * 配置权限。比如将权限 minecraft.command.* 分配给玩家，那么玩家将获得 minecraft.command 这个节点下的所有权限，也就是说玩家将可以使用 Minecraft 原版的所有命令。而将 quickshop.* 权限分配给玩家则可以让玩家使用 QuickShop 插件提供的全部内容，不受任何权限限制。

LuckPerms 采用网页编辑器的方式配置权限，免去了直接调整配置文件的麻烦。在游戏或控制台中运行命令 lp editor（与 luckperms editor 效果相同，lp 是 luckperms 的命令简写）。并将返回的链接复制到浏览器中，即可打开网页编辑器。

LuckPerms 插件在打开网页编辑器时会自动读取所有插件的权限，你也可以直接在其中查看其他插件的权限。

LuckPerms 以权限组（Group）的方式来管理权限。每个权限组中包含若干个权限。通过将权限组分配给玩家，便可以给玩家赋予组中的权限。

每个新玩家在进入游戏时都默认被分配了 default（译为默认）权限组。你也可以在网页编辑器中创建新的组，并通过命令的方式将这个组分配给玩家，例如额外创建一个名为 admin 的组，并给这个组分配可以使用所有原版命令的权限：

权限插件基本是不可缺少的，没有合理的权限管理的服务器本身是充满隐患和缺乏阶段性的。

#### 经济插件

基本所有类型的服务器都会拥有自己的经济系统。在技术上，一般表现为每个玩家都有自己的虚拟账户，其中储存了每个玩家剩余的虚拟货币数量，以便玩家进行消费。

最初，经济插件有很多，每个插件都用自己的方式储存玩家的剩余金额。这造成了兼容性的问题：一个其他的插件想要利用这种经济系统（比如箱子商店插件，需要依据买卖行为修改玩家的余额），则必须为所有类型的经济插件提供进行适配，因为这些插件不会知道，服主到底在自己的服务器中安装了哪一个经济插件。随着经济插件的种类增多，其他插件作者的压力当然也会越来越大。

于是，插件作者们就开发了一套标准，并将其设计为一个统一的前置插件，名为 [Vault](https://www.spigotmc.org/resources/vault.34315/)。一切经济插件都需要使用这个前置插件提供的方法储存经济数据。这样，当其他插件想要修改玩家的余额时，就不再需要研究每一个经济插件的储存方式，而只需要与他们一样使用这个 Vault 插件提供的方法修改玩家的余额即可。

因此，我们也就明白了，当我们想为自己的服务器引入一个经济系统时，基本都需要同时安装两个插件：一个是你选择的经济插件，另一个则是 Vault。

常见的经济插件可以分为以下几种类型：

- 虚拟经济，玩家的货币是没有实体的，以纯数字的形式储存在玩家的虚拟账户中
- 物品经济：玩家的货币有实体，可以被玩家拿在背包中。背包中有的货币物品数量就是玩家的虚拟账户余额。当玩家消费时，背包中的货币物品也会被消耗
- 混合经济：玩家的货币可以被提取成物品，也可以被再次存入虚拟账户变成一串数字

其中的代表插件则有：

- 虚拟经济
  - [EssentialX](https://www.spigotmc.org/resources/essentialsx.9089/) - 经济模块
  - [Xconomy](https://www.spigotmc.org/resources/xconomy.75669/)
- 物品经济
  - [The Gold Bank](https://www.spigotmc.org/resources/the-gold-bank.70562/)
  - [MMOEconomy](https://www.spigotmc.org/resources/mmoeconomy.112599/)
- 混合经济
  - [The Piggy Bank](https://www.spigotmc.org/resources/the-piggy-bank.74104/)
  - [CMI](https://www.spigotmc.org/resources/cmi-298-commands-insane-kits-portals-essentials-economy-mysql-sqlite-much-more.3742/) - 经济模块（有简单的支票功能，所以被归为混合经济）

对于虚拟经济，大部分服主都会选择使用如 EssentialX 或 CMI 这类基础插件自带的经济模块，而不是额外安装一个经济插件（更节省性能、更方便管理等）。

对于物品经济，因为其开发难度较大，所以现存的插件数量较少，且质量较为参差。

你可以根据自己服务器的设计选择适合自己的经济插件。

当然，经济插件也不是必要的。经济系统本身会为服务器所有者带来更大的管理压力，他们需要花费更多的精力来保证经济系统不会崩溃（严重的通货膨胀是典型问题）。而有时一个多余的经济系统也无法给服务器本身带来高于成本的回报。

#### 领地插件

大部分与生存相关的服务器都会安装领地插件。领地插件可以让玩家保护自己的建筑、物品等资产不被他人侵犯。

领地插件的核心是圈地机制与权限控制机制。

圈地机制是将一个物理范围纳入玩家的领地的机制。一般表现为玩家可以手持某种工具，点击两个方块来定义一个长方体，之后通过命令等方式将这个长方体范围划为自己的领地。

权限控制机制则是对玩家的领地进行控制的机制。一般表现为玩家可以通过一些命令或者简单的菜单来控制哪些行为不可以在领地内发生，比如非领地的成员不可以打开领地内的箱子等容器、领地范围内不能刷新怪物等。这个机制一般与职位机制（领地可以为玩家定义职位，比如访客、成员、管理员、主人等）一起存在，方便领地所有者更好地管理自己的领地。

在以上两个核心机制的基础上，各类领地插件都进行了自己的创新。创新的方向主要是以下几个：

- 给领地本身新的定义：将领地本身分级，比如 [Towny Advanced](https://www.spigotmc.org/resources/towny-advanced.72694/) 插件将领地分为国家和城镇两级，国家领地可以管理城镇领地，城镇领地可以升级为国家领地
- 创新圈地的方法：比如 [Lands](https://www.spigotmc.org/resources/lands-%E2%AD%95-land-claim-plugin-%E2%9C%85-grief-prevention-protection-gui-management-nations-wars-1-20-support.53313/) 插件以区块而不是方块作为最小单位定义领地；Origin Realm 服务器的自制领地插件让玩家设置一个中心箱子，玩家在其中投入资源就可以以箱子为中心扩展领地范围等
- 给领地添加旁支功能：如 [KingdomsX](https://www.spigotmc.org/resources/kingdomsx.77670/) 插件为领地添加了防御塔、守卫等与「保护」相关的功能

常见的领地插件有如下几个：

- [Towny Advanced](https://www.spigotmc.org/resources/towny-advanced.72694/)
- [Lands](https://www.spigotmc.org/resources/lands-%E2%AD%95-land-claim-plugin-%E2%9C%85-grief-prevention-protection-gui-management-nations-wars-1-20-support.53313/)
- [KingdomsX](https://www.spigotmc.org/resources/kingdomsx.77670/)
- [Residence](https://www.spigotmc.org/resources/residence-1-7-10-up-to-1-20.11480/)

当然，领地插件也不是必要的。失去保护自己资源的能力有时也可能称为刺激的来源。著名的 2B2T 模式就包含了这种思想（「无政府主义」服务器，核心要素是无反作弊，风格是混乱无序，玩家以互相伤害和搞垮服务器为目标进行游玩）。

#### 菜单插件

菜单插件可以为服务器添加各类菜单，帮助玩家更好地与服务器内的各类功能交互。

插件服的菜单界面实际上是用箱子实现的。菜单插件可以为玩家打开虚拟的、装了不可取出的物品的箱子界面。当玩家点击箱子中的物品时，插件可以监听这个事件并根据点击的按钮不同从而执行不同的操作。常见的操作有：执行命令（免去玩家记忆命令的麻烦）、打开另一个插件的菜单等。

菜单插件基本都具有高度的自定义性。服主可以控制菜单的行数、菜单的布局、菜单中物品的位置、点击物品执行的操作等，是配置难度最高的插件类型之一。近年来，以 Trmenu 为首的菜单插件还与脚本语言（JS、Kether）结合，意图让服主自行编写功能，从而替代某些小型插件（比如编写一个简单的商店、为 RPG 服务器编写一个锻造系统等等）。

常用的菜单插件有很多，以下是一些例子：

- [TrMenu](https://github.com/TrPlugins/TrMenu)（现已停止更新，GitHub 中仍能下载到最后的版本，支持 1.20.1）
- [CommandPanels](https://www.spigotmc.org/resources/commandpanels.67788/)
- [DeluxeMenus](https://www.spigotmc.org/resources/deluxemenus.11734/)

菜单插件同样也是按需使用的。多余的菜单反而会让玩家更加难以理解你服务器的内容。

#### 材质相关插件

材质插件是近几年提出的新概念，也是插件服界主要的技术革命之一。

插件没有向游戏注册新物品、新方块和新生物的能力。以往，服主通常使用：

- 给相同材质的物品不同的命名并为其添加各种属性和技能，假装这是一个新物品
- 给相同的生物不同的命名并为其添加各种属性和技能，假装这是一个新生物

这种模式正式被打破是在 Origin Realm 服务器诞生之后。这个服务器创新性地大规模使用材质包来添加新物品、新方块和新生物，开创了插件生存服材质包化的先河。

这类插件可以根据简单的配置文件而生成材质包并将其附加到物品、方块和生物上，并在玩家进服时自动为它们安装生成的材质包，从而为服务器添加全新的内容。

添加新内容的技术原理可以抽象的理解为：

- 物品：每个物品都有一个模型 ID，不同的模型 ID 可以绑定不同的材质。比如，一把石剑的模型 ID 是 1，我们将圣剑的材质绑定到石剑的模型 ID 2 上，之后，再生成一把拥有 ID 2 的石剑。尽管它在技术上仍是石剑，但在安装了材质包的玩家眼里，它已经是圣剑了
- 方块：与物品的逻辑相似。屏障、线、展示框、音符盒、刷怪笼这几类方块在技术上可以拥有大量不同的「状态」（比如音符盒在游戏中可以有不同的音调和乐器类型，排列组合可以得到数百种状态），而我们可以将不同的材质绑定到这些方块的不同状态上。比如，我们将一个蓝色砖块的材质绑定到音符盒的**贝斯音色、0.5 音高**状态。接着，由插件向玩家分发一个拥有特殊 NBT 标签的音符盒。当玩家放下这个音符盒时，插件检测到对应的 NBT 标签并根据标签设置这个音符盒的音色和音高。这样，技术上这个方块虽然仍是音符盒，但在安装了材质包的玩家眼里，它已经是蓝色砖块了
- 生物：插件会根据提供的生物模型文件生成大量隐形的盔甲架（老版本）或显示实体（新版本），并控制这些实体的旋转、移动等以达到模拟生物动作的目的。同时，这些实体也都拥有特殊的内部 ID，材质包可以将不同的材质附加到这些特殊实体上。这样，虽然在技术上，四处移动或进行攻击的仍然只是一些看不见的内部实体，但在安装了材质包的玩家眼里，它们已经是一个完整的生物了。

除了这些新内容的添加之外，材质包还可以更加专注于外观的改善，例如可以为菜单添加背景，改变以往千篇一律的箱子界面（虽然在技术上仍是箱子界面）、可以添加在聊天栏中使用的表情包（虽然在技术上被发送的只是一些特殊的 Unicode 字符）等。

总而言之，这些新技术都有「以假乱真」的能力，这也是插件服玩法拓展的核心思想。

有了这些新技术，插件服的玩法生态和外观也就开始变得逐渐丰富和精致起来了。

常见的材质包类插件有如下几个：

- [ItemsAdder](https://www.spigotmc.org/resources/%E2%9C%A8itemsadder%E2%AD%90emotes-mobs-items-armors-hud-gui-emojis-blocks-wings-hats-liquids.73355/)：基本可以添加所有类型的新内容
- [ModelEngine](https://mythiccraft.io/index.php?resources/model-engine%E2%80%94ultimate-entity-model-manager-1-19-4-1-20-4.1213/)：专攻添加新生物，没有自动分发材质包的功能

### 插件服的设计

在上文中，读者了解了插件服常用的几类核心插件以及它们的作用。之后，我们就要开始利用这些插件规划和设计自己的服务器。

归根结底，设计一个服务器与设计一个游戏是一样的，除了技术外，文学、美术等等素养也十分重要。故从本节开始，文章内容将会由客观变得主观，由具体变得抽象。

#### 插件服的特点

想要获知插件服的特点，自然要与其他两种类型，即模组服和原版服进行比较。

与模组服相比，插件服具有如下特点：

- 玩家无需安装特制客户端即可进入服务器
- 易于扩展（在插件能做到的范围内，对服务器的修改比模组要简单很多）
- 对服务器主机和玩家电脑设备的性能要求低
- 无法添加新内容（需要借助「以假乱真」的技术）

与原版服相比，插件服则有如下特点：

- 可以对原版进行扩展，却又不过分脱离原版

因此我们可以发现，想要设计一个好的插件服，就是要充分利用其易于扩展的特点。

#### 常见的插件服

目前，市场上的插件服分有以下几类典型：

- 生存扩展类
  - 岛屿系：包括空岛、酸岛和木筏求生等，玩法围绕着玩家孤立的岛屿展开
  - 起源（Origin Realm）系：主要玩法仍是原版玩法（挖矿、建筑、生存），但大量使用材质包添加新内容和装饰性玩法（如种植、烹饪等）
  - 城镇经营系：核心玩法是建立和经营城镇，有时还包含 PVP 内容（扩展势力范围、城镇械斗等），在原版玩法的基础上构建
  - 其他系：围绕一个迷你的核心玩法或概念构建，如 Life Steal（玩家可以偷取他人的最大生命）、极限生存（每个玩家只有一次游玩机会）、近视生存（玩家的视距极小）等，同样依托于原版玩法构建
- RPG 类
  - 传统系：符合大部分人印象的 RPG，玩家扮演某种角色，在某个特定的世界观下冒险，关键词有怪物、遗迹、武器词条、职业、升级等等，其中较为不典型的有监狱风云等
  - 社会模拟系：新兴的 RPG 类型，几十到上百个玩家在仅持续数周的短暂时间中被分为数个组，围绕一个主题进行生存和争斗。特点是单周目持续时间短、封闭（一个周目内一般不会允许中途加入太多新人）、高度沉浸（较短的持续时长让服务器作者可以专注于世界观和核心玩法的构建）和充满活力
- PVP 类
  - 塔科夫系：类似游戏《逃离塔科夫》，关键词是仓库、永久死亡（死亡清空背包）、逼真的生存物理等
  - 竞技场系：玩家围绕着某一个规则，在限定的范围内以固定的模式进行 PVP，典型的有职业战争与 PVP 练习等，感觉上有些类似于小游戏类
  - 帮派战争系：玩家组成帮派，以摧毁对方为目的进行 PVP，在原版玩法的基础上构建
  - 无规则系：纯粹的 PVP，典型的有 2B2T 模式
- 小游戏类：由大量多人或单人小游戏组成的服务器，典型的有 Hypixel
- 其他类
  - 创造服：大多以自由建筑为核心，关键词是是受限的创造模式、地皮、建筑等等
  - 主播服：玩法不是重点，谁开的才是重点，意在促进粉丝交流、创造视频素材等
  - 混合系：将上述各类各系典型服务器的玩法排列组合而成的服务器

最初，缺少游戏设计经验的服主一般都会做出不完整的起源系或城镇经营系服务器。在这个阶段，服主们仍在扩展自己的插件阅历，会不自主地将自己感兴趣的插件一股脑地塞进服务器中。

若没有一定的玩家基础，这类服务器一般都会以失败（缺少新鲜血液、玩家一家独大、通货膨胀无法抑制，被迫换周目等）告终。

这类服务器的问题在于，服主并没有用心地进行设计，只是走一步看一步、凭一己之见行动。在服务器走向衰落的过程中，负责任的服主可能还会落入自我怀疑和失去主见，即由仅凭一己之见变为仅想满足玩家的需求，并认为只有听从玩家的喜好修改服务器才能拯救它。

实际上，设计插件服需要服主有「制作游戏」的觉悟。这个过程与拍摄电影类似，需要工作者提前做好剧本（游戏设定）并围绕着设定逐步展开工作。

那么自然有人要说，这是一种一家独大的思维，没有关心玩家的需求。

问题在于，关心玩家的需求，吸收玩家的建议，并不意味着把玩家当作上帝，并遵循他们的一切建议。玩家是比服主更不专业的游戏设计者，也不了解服主本人对服务器的安排和设计。同时，玩家的建议也有强烈的主观色彩，且往往视野范围较小。玩家建议的价值在于为服主提供参考，从庐山之外观察庐山，发现服主视角下不易察觉的问题。而不是操控服主的行动乃至直接干预服务器本身的设计。

举个极端的例子，如果在一个 PVP 服务器中，一个玩家为了想要快速压倒其他玩家而提出开设充值购买强力武器的系统，难道服主就要为了他一个人这么做吗？这种系统存在于 PVP 服务器中，显然是有违游戏公平性的。

作为服主，我们需要对自己设计的玩法有信心，并且秉持将其贯彻到底的觉悟，才能设计好一个服务器。

若仍担心自己设计的玩法是脱离玩家的，也可以采取提前发布玩法白模的方式，让玩家可以提前体验你的设计并提出改进建议。所谓玩法白模，就是去除一切装饰性内容的纯粹的玩法，例如《超级马里奥兄弟》这类横板闯关游戏的白模就是一个白小球可以被玩家操控着在屏幕上移动，碰到其他移动的红色小球就会死亡。而一个传统 RPG 系的服务器的白模就可以是仅有寥寥几种典型的武器和三四个属性词条的服务器本体。

打好了思想基础之后，让我们回到上文列举的五类服务器中，讨论其设计思路和优劣。

首先是生存扩展类。「Minecraft 不是游戏引擎，其本身就具有足够的魅力吸引玩家游玩」。坚信这一点的服主往往会将设计向这类服务器靠拢。比起把游戏本体改的花里胡哨，这类服务器更注重挖掘原版的魅力，并在其上做出增强。例如岛屿系服务器注重在多人游戏中营造单人游戏的环境、起源系服务器注重装饰游戏本体、城镇经营系服务器注重建立玩家社区，让玩家自己产出新内容。

成功的这类服务器通常可以营造优质的玩家氛围，并维持较长的生存周期。所谓「不删档」、「不关服」等词汇一般也被用于形容这类服务器。

这类服务器的问题在于，同质化较为严重。同时，相对低廉的制作成本让市场上充斥着质量参差不齐的此类服务器。这类服务器想要脱颖而出，不外乎在核心玩法上推陈出新，抑或是稳定性极高、平衡性极好、玩家社区素质极高等。

其次是 RPG 类。有一定游戏设计和开发能力并认为 Minecraft 有足够的能力成为游戏引擎的服主往往会选择这条道路。这类服务器遵循龙与地下城式（DAD，Dungeons & Dragons）传统 RPG 的设计思路，向游戏内添加大量武器、属性词条、新怪物、遗迹、魔法、技能等内容，意图让玩家为了战胜更强大的怪物而醉心于提升自己的战斗力。同时，近年来开放世界 RPG 游戏的大热也让这类服务器开始回归原版的开放性（家园系统等）。

这类服务器通常是可以创造最多收入的服务器，也是让玩家玩的最「爽」的服务器。同时，它也是快餐服之争的核心。

这类服务器的问题在于，超长的设计周期可以让服主花费极长的时间在制作环节，以至于让原本积攒的玩家都跑的没了影。更有可能的是，服主甚至无法坚持完成服务器，只能中途放弃。

接着是 PVP 类。我了解的制作 PVP 类服务器的服主大多是 PVP 玩家出身，了解 PVP 的乐趣且明白玩家的 PVP 需求和 PVP 圈的各种「潜规则」。这类服务器本身的技术创新较少，近年来新兴的逃离塔科夫式 PVP 服务器也没有得到足够的推广（主要是开发难度较高和脱离 Minecraft 的核心玩法）。

这类服务器的问题在于玩家基础较差。这类服务器想要成功开设，需要服务器制作者本身自带一定数量的玩家基础。

之后是小游戏服务器。小游戏服务器可谓是现存数量最少的服务器类型之一。原因主要是小游戏服务器的乐趣在于庞大的玩家数量带来的不确定性。因此，玩家更愿意前往人多的小游戏服务器游玩，这也就使 Hypixel 成为小游戏服务器的垄断者。

当然，小游戏服务器本身也具有高于原版扩展类服务器的开设难度。为了支撑庞大的玩家数量和多实例化的游戏地图，一般需要利用负载均衡等技术。

最后是其他类。其中值得一提的是创造服。创造服的市场存量也较少，这是因为大部分创造服给玩家的体验无限接近单人模式。成功的创造服大多是主播服（依旧是需要自带玩家基础）。

至于其中的混合系，其实是大部分服主会选择的道路。明白创新的重要性的服主最常采用的创新方式便是不同类型服务器玩法的缝合。这种创新方式实际上风险较高，容易导致失去特色和方向的问题，需要提前做好规划。

想要在如上繁杂的服务器类型中进行选择，服主可以参考以下几点：

- 兴趣：兴趣永远是第一要素。服务器的制作过程是漫长的，维护过程是令人痛苦的。若没有兴趣，服主难以在这种过程中坚持到最后。
- 能力：不同类型的服务器，其开发难度是不同的。一个没有经验的服主和服务器开发团队一开始就选择制作 RPG 类服务器是不现实的。当然，也不要忘记能力是可以培养的。
- 市场：玩家市场青睐的服务器类型是时常变化的。一般，迎合市场需求的服务器是更容易招揽更多玩家的。当然，也不能忘记服主和玩家并只不是服务与被服务的关系，而也是双向选择的关系。实际上，所谓的市场趋势恰恰是改变市场的人创造的。

#### 如何设计插件服

在了解所有类型的服务器的基本特点并选择自己的开发方向之后，我们需要开始设计服务器。

设计这事，千人千面。比起阅读这篇文章并试图学习设计，你更应该去阅读游戏学领域关于游戏设计的著作，如《体验引擎》（泰南·西尔维斯特）、《快乐之道》（拉夫·科斯特）等。下文只讨论本人对设计 Minecraft 服务器区别于设计游戏的部分的看法。

本人主要制作传统系 RPG 服务器，在设计上提倡维护原版风格和最小化玩法这几个原则。以下是对这两个词汇的解释：

- 维护原版风格：所谓原版风格，可以理解为原版美术风格和原版玩法风格。Minecraft 终究不是游戏引擎，我们的服务器中的玩家大多也是先接触原版再游玩服务器。维护原版风格是服主应该尽的义务之一，为的是尊重官方的劳动成果。美术风格方面，主要表现为添加的新界面、新物品、新生物等的外观放在原版中是否显得突兀。这也导致了本人对在服务器中引入传奇风格的界面和武器装备这类行为十分厌恶。玩法风格方面，主要是维护原版的几个核心要素，包括挖矿、合成与建筑等。
- 最小化玩法：没有核心玩法的服务器是散漫的，是让玩家抓不到重点的。所谓没有核心玩法，大多是由玩法过多造成的。如上文曾说的，经验不足的服主喜欢将玩法一股脑地塞进服务器，这就使玩法扩大化。因此，最小化玩法即使将玩法进行精炼，与服务器核心设计强耦合，此举可使新玩家的学习曲线更加平缓，也可以使你的服务器给人和谐统一之感，更可以节省服务器性能。

### 关于电子游戏作为新媒体的价值和其艺术化的论述

电子游戏在我国处于社会批判的风口浪尖。这种批判在 2018 年 [江苏少年坠楼案](https://www.163.com/dy/article/IA1DAOIB055646T8.html) 之后更是盛极一时，达到了前所未有的激烈地步。著名的「电子鸦片论」也是在这个时期产生的。

无独有偶，电子游戏行业在美国也遭受同样的指控，而这种指控同样在 1999 年 [哥伦拜恩高中枪击案](https://zh.wikipedia.org/zh-cn/%E7%A7%91%E5%80%AB%E6%8B%9C%E6%A0%A1%E5%9C%92%E4%BA%8B%E4%BB%B6) 后达到顶峰。

对电子游戏的指控无外乎以下几种：

1. 电子游戏是纯粹为了娱乐而产生的，没有为社会创造价值
2. 电子游戏会传播暴力的思想
3. 电子游戏会消磨青少年的精神

对于第 1 点，答案是很简单的。电子游戏在表现上与电影相似，由音像、文学、美术等多种媒体组合而成，又在其上产生了自己的表现逻辑，可以说是典型的新媒体。大部分电影都是为了娱乐，大部分美术作品也是为了娱乐，流行音乐则更是为娱乐而生。娱乐本身就是媒体的固有属性之一，并不能因为当前电子游戏展现出的多是其娱乐作用部分就否定电子游戏的其他意义。

戏剧在诞生之初被上层人士所不齿，小说在一开始是一群太太的消磨时间之物，电影更是为了逃避现实而生。但随着其中蕴含的思想和遵循的系统形式越发加深和完善，它们现在被人尊称为艺术。这些媒体为传播信息和共享体验而生，而一旦这种信息和体验变得需要思考才能被提取出来，艺术便产生了。电子游戏作为现代的新媒体，显然也将与这些媒体类型共享相似的命运。

那么对于 2 3 两点，则也可进行驳斥。游戏与其他艺术类型一样，都是片面的形式系统。在约百年的电子游戏发展史中，电子游戏看似快速发展，但剥开其美丽的外壳（美术、音像等），其游戏系统（又称玩法）本身并没有多少创新。可以说，游戏并不是依靠游戏系统，而是依靠外表向用户传递信息的，这也与其他艺术形式相同。

若我设计一个游戏，玩家扮演战败前的集中营中的纳粹士兵，他们在地上挖了一个深坑，在其下放满致命毒气，将犹太人排成一列，并将他们逐个推下去。玩家的游戏目标是高效率地处理掉这些有胖有瘦，有高有矮的犹太人。若将他们推下去的落点合适，使他们整齐地堆成一堆，最下面的犹太人就会被毒气毒死，而若没有排列好这些犹太人，它们就会逃出来。这款游戏一定是宣扬暴力的，是被社会所不齿的。但它其实就是俄罗斯方块改编而来。

看来，2 3 点所指控的罪责并不属于电子游戏这一媒体本身，而是属于电子游戏的设计者们。他们设计与传达的内容确实给人带来了快乐，但是也在电子游戏的体验中将错误的技能和印象传达给人们。

那么作为电子游戏的设计者，理应规范自己的思想，正视自己的行业，重新思考电子游戏能为社会带来什么。这种由社会责任感引起的思考就促进了游戏学的产生（在中国仍未形成体系）。
